System Architecture

High-Level Architecture

NestJS Feature Modules

  • SyncModule — orchestration, dedup, approval
  • MessagesModule — WhatsApp + Gmail, storage
  • LlmModule — Gemini parser + embeddings
  • CalendarModule — events, Google sync
  • AuthModule — OAuth 2.0 (PKCE)

Storage

Single SQLite file at ~/.config/parentsync/parentsync.db. TypeORM with synchronize: true — schema auto-applied on every startup.

Runtime

Electron embeds NestJS backend in the main process. Frontend runs in the renderer. IPC bridge via contextBridge preload.

Stage 1 — Ingestion

Message Ingestion: WhatsApp & Gmail

WhatsApp (whatsapp-web.js)

  • Chromium session managed by whatsapp-web.js (embedded, no external Chrome)
  • User authenticates via in-app QR code on first launch
  • Channels & group chats monitored per-child settings
  • Messages fetched at user-configured hour(s) via NestJS cron
  • Images extracted (up to 4 MB per message, stored as base64 JSON)
  • De-duplication pass runs before LLM (Phase 20)

Gmail (Gmail API, OAuth 2.0)

  • Paginated messages.list with search query per-child label
  • Searches Inbox, Spam, Trash (since Phase 1.1 fix)
  • Raises default limit; deduplicates by message-ID
  • Same downstream pipeline as WhatsApp from this point

Why two dedup columns?

contentHash — O(log n) indexed lookup for exact or near-exact matches.
embedding — cosine-similarity scan for paraphrased/forwarded duplicates. The hash is the fast path; the embedding handles reformulations.

Lazy population

Both embedding and contentHash are NULL until the message is first seen by MessageDeduplicationService. This means older messages have NULL until the next sync runs over them.

Stage 2 — Grouping

Proximity Grouping (2-Hour Window)

Before sending messages to the LLM, EventSyncService merges consecutive unparsed messages from the same channel that are within 2 hours of each other. This reduces API calls and lets the LLM see thread context.

How it works

  • Sort unparsed messages by channel + timestamp
  • Start a new group whenever gap to previous message > 2 hours
  • Within a group: concatenate all content strings
  • Group carries a childId (from first message)

Result: GroupMeta[]

Each entry: { messages: Message[], mergedContent: string, channel, childId, indices }.
This array feeds the dedup stage, then the LLM batch call.

Concern

The 2-hour window is hardcoded. A slow-drip thread spanning the day will produce many small groups, causing more LLM calls.

Stage 3 — Semantic Deduplication (Phase 20)

Semantic Deduplication: SHA-256 + Gemini Embeddings

Fail-open contract

findDuplicateOf() never throws. All error paths return { match: null } — the group is treated as fresh. Avoids blocking the sync on a transient Gemini outage.

DedupResult shape

{ match: DedupMatch | null, contentHash: string, embedding: number[] | null }

The result is passed back to EventSyncService which stamps both columns onto all messages in the group.

Parameters

  • LOOKBACK_DAYS = 30
  • LOOKBACK_LIMIT = 1000
  • Default threshold = 0.92 (user-configurable)
  • Embedding dim = 768 (Gemini)

⚠ O(n) linear scan

No HNSW or ANN index. SQLite has no native vector ops — cosine is computed in JS over the 1000-candidate window. Will degrade if the candidate set grows.

Prompt context (each call)

  • System instruction: extract events, tasks, dates
  • Negative examples pool (up to 50 items, from 😢 reactions)
  • Customizable prompt suffix (user-editable in Settings)
  • All fresh group texts concatenated, delimited by group index

ParsedEvent fields

  • title, date, time?, location?
  • syncType: 'event' (timed) | 'task' (date-only)
  • source, sourceId, sourceContent (snapshot)
  • childId, calendarColorId

⚠ Parser cache risk

24h cache keyed by content + prompt-version hash. If the user changes the custom prompt, existing cached entries with the old prompt-version key expire naturally — but within the TTL window, new events won't reflect the updated instructions.

Stage 5 — Event Creation

Event Creation & 3-Layer Deduplication

Three independent dedup mechanisms prevent the same real-world event from appearing twice.

↓ no exact match

↓ confirmed unique
INSERT calendar_events row

createEventsInTransaction()

  • Wraps all inserts + message parsed=true stamps in one DB transaction
  • On any error: entire transaction rolls back, messages stay unparsed (will retry next sync)
  • Stamps embedding + contentHash onto message rows here
  • Increments metric.events_created_total

⚠ LLM dedup retirement

OLD-2 (detectDuplicateOfExisting) was a temporary bridge added in Phase 16. The plan is to remove it once Layer 1 (Phase 20) accumulates enough data. A 4-week review was scheduled by calendar reminder — tracked manually, not in code.

⚠ Metrics are point-in-time

metric.events_created_total and metric.event_dedup_llm_fires are stored as key-value rows in the settings table. No time series — you can't chart trends over time.

Stage 6 — Approval Routing

Approval Routing Decision

approvalMessageId, set approvalStatus = PENDING Awaiting 👍 / 😢 reaction…

sendForApproval() details

  • Formats a human-readable message with event details
  • Generates an ICS calendar attachment
  • Sends to the configured WhatsApp approval group
  • Stores the sent message's ID in approvalMessageId

Direct sync cases

  • No approval channel configured in Settings
  • Event date is in the past (can't approve retroactively)

In both cases: approvalStatus = 'NONE', event synced to Google immediately.

⚠ WhatsApp session expiry

If the WhatsApp session expires while events are PENDING, approval messages can't be sent and reactions can't be received — silently. No fallback timeout or notification to the user.

Stage 7 — User Reactions

User Reactions & ApprovalStatus State Machine

What triggers state changes

  • PENDING — sent to WhatsApp approval group
  • APPROVED — 👍 emoji reaction on the approval message
  • REJECTED — 😢 emoji reaction on the approval message
  • Undo — removing the reaction reverts to PENDING

On APPROVED

Event is synced to Google Calendar (or Google Tasks for date-only events).

On REJECTED

Captures a NegativeExample from sourceContent snapshot. The LLM prompt is enriched with this example on the next sync call.

⚠ Reactions can arrive in any order, any time

The system handles out-of-order reactions (approve then undo then approve), but there is no TTL or expiry for PENDING events — they can accumulate indefinitely.

sourceContent importance

This is the exact merged-group text the LLM saw, captured at parse time. When the user reacts 😢, this content becomes the NegativeExample body — so the LLM learns from exactly what it was shown, not a later version.

Idempotent Google sync

If googleEventId is already set, the sync service updates rather than inserts. syncedToGoogle is the quick-check flag to skip already-synced events.

Feedback Loop

NegativeExample Feedback Loop

approvalMessageId Read sourceContent snapshot Save NegativeExample { content: sourceContent, createdAt, reason } Pool management Cap = 50 · if full → drop oldest entry Next LLM call includes NegativeExamples in prompt LLM is told: "do NOT extract events like these" feedback

Purpose

When the user rejects an event (😢), it means the LLM extracted something it shouldn't have. The NegativeExample captures what was in the message and feeds it back as a "do not do this" example in subsequent prompts — improving precision over time.

⚠ Silent truncation at 50

The pool is capped at 50 entries. When the cap is reached, the oldest entry is dropped silently. There is no UI indicator that older negative examples have been discarded — the user can't know that the LLM has "forgotten" a rejection.

Undo-reaction cleans up

If the user removes their 😢 reaction (undo), the event's approvalStatus reverts to PENDING. The NegativeExample that was captured is not automatically removed however — a minor inconsistency.

Resilience

Error Handling & Resilience

Fail-open dedup

MessageDeduplicationService.findDuplicateOf() wraps all operations in try/catch. On any error (network, API timeout, DB error), it returns { match: null } — treating the group as fresh. Sync never blocks on a dedup failure.

Transactional event creation

createEventsInTransaction() runs inserts + message stamps in a single SQLite transaction. Any failure rolls back — messages remain parsed=false and will be reprocessed on the next cron tick.

AppErrorEmitterService → SSE → Frontend

Blocking failures (OAuth expired, LLM API key invalid) are emitted as SSE events to the frontend, which renders an ErrorModal with a user-actionable message. This prevents silent failure for critical errors.

Retry model

  • No in-process retry logic for LLM calls — failures propagate to the transaction rollback
  • Natural retry = next cron tick (user-configured interval)
  • WhatsApp reaction listener is event-driven (no polling)

⚠ No dead-letter queue

If a group of messages permanently fails LLM parsing (e.g., malformed LLM response), the messages remain parsed=false indefinitely and will be reprocessed on every sync — including in the dedup cost.

Graceful shutdown

NestJS app.enableShutdownHooks() — ensures in-flight WhatsApp sessions are closed cleanly. Electron main process forwards SIGTERM to the embedded backend.

⚠ TypeORM synchronize: true

Schema changes auto-apply on every app start via TypeORM's synchronize: true. Safe for a single-user app, but any column drop or rename would silently destroy data. There are no migration files to inspect.

Architectural Review

Architectural Concerns ⚠️

⚠️
1. O(n) embedding scan — no ANN index
Cosine similarity is computed in JS over up to 1,000 candidates per sync. No HNSW/FAISS index. Will degrade linearly as the message volume grows past the lookback window. Consider sqlite-vss or a separate Chroma/Qdrant sidecar for Phase 21+.
⚠️
2. Metrics stored as key-value, not time series
metric.events_created_total and metric.event_dedup_llm_fires are counters in the settings table. You cannot chart trends over time or see when activity spikes occurred. A metrics table with (key, value, timestamp) rows would cost ~1 KB/day.
⚠️
3. Images column — up to 4 MB per message row
WhatsApp images are stored as base64 JSON in the messages table. This inflates the SQLite file and can spike memory during ORM hydration. Consider moving images to the filesystem and storing only a path reference.
⚠️
4. WhatsApp session expiry — no fallback
When the whatsapp-web.js session expires, approval messages can't be sent and reactions aren't received — silently. PENDING events accumulate. There's no session health check, fallback notification, or TTL on PENDING events.
⚠️
5. TypeORM synchronize: true
Fine for a private single-user app, but risky for any future multi-user migration. Column drops and renames will silently destroy data. No migration history exists to audit schema evolution.
⚠️
6. Parser cache masks prompt changes (24h TTL)
The LLM response cache is keyed by content + prompt-version hash. Within the TTL window, cached entries won't reflect a changed custom prompt — the user might not see the effect of their prompt edits for up to 24 hours.
⚠️
7. NegativeExamples cap=50, truncation invisible
Once the pool reaches 50, older negative examples are silently dropped. There's no UI to show the pool size, review examples, or confirm they're still influencing the LLM. The undo-reaction path also doesn't remove the captured example.
⚠️
8. LLM event-dedup (Layer 3) — no automated retirement
detectDuplicateOfExisting() was a temporary Phase 16 bridge. Retirement is tracked only by a personal calendar reminder, not in code. If Phase 20 semantic dedup proves effective, Layer 3 should be removed in a planned Phase 21 cleanup.

Key design choices

  • Fail-open dedup — never blocks sync on embedding failures
  • Batch LLM call — single API request for all fresh groups
  • Transactional event creation — rollback on any failure
  • sourceContent snapshot — correct NegativeExample capture
  • SHA-256 fast path — O(log n) before O(n) embedding scan

Top concerns to address

  • ⚠ O(n) embedding scan — add ANN index
  • ⚠ WhatsApp session expiry — add health check + PENDING TTL
  • ⚠ Images in DB — move to filesystem
  • ⚠ Retire Layer 3 LLM dedup (Phase 21)
  • ⚠ Add time-series metrics table

  • B-tree index on contentHash
